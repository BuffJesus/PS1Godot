### Kenney figure texture is broken — re-diagnose, was NOT a mesh issue

User screenshot at `C:/Users/Cornelio/Pictures/current/figure.png`
shows the Kenney character in T-pose, mesh complete, with the
material override applied — but the **texture sampling is wrong**:

- **Face** shows correct skin tones (eye visible, brown hair area).
- **Right hand** shows correct skin tones.
- **Body, head sides, head back, legs** show a chaotic mosaic of
  gray/white triangles. Not solid gray — fragmented patches.

Earlier we assumed this was missing faces (backface cull) and
patched the renderer to skip the cull on skinned tris. The cull
patch is still correct (mesh now visibly complete), but the **real**
problem is texture/UV-related.

**Suspect causes — investigate in order:**

1. **VRAM page-boundary sampling.** PSX GPU samples within a 256×256
   page. If the skin texture was packed at an offset such that some
   triangles' UVs sample past the texture's actual region, those
   triangles read garbage from adjacent VRAM (CLUTs, other textures).
   The "fragmented patches that look like other textures" pattern is
   exactly this. Check the export log for the skin texture's
   `tpage`/UV remap, and compare the atlas position against the
   skin's UV bounding box.

2. **4bpp quantization collapsing distinct colors.** humanMaleA.png
   is being packed at `BitDepth=0` (4bpp = 16 colors). If the
   quantizer picked 16 colors that don't separate skin from clothes
   from background, large adjacent UV regions can land on the same
   palette index → looks like solid gray patches. Test: temporarily
   set `BitDepth=1` (8bpp = 256 colors) on the skinned mesh,
   re-export, see if the body looks coherent. If yes, the fix is
   either pre-quantize the source PNG or move skin to 8bpp and trim
   another texture to balance VRAM.

3. **Mirrored / out-of-range UVs.** Character models often use
   mirrored UVs (left and right halves sample the same region with
   one side using `1.0 - u`). If the FBX exports UVs outside [0,1]
   or with negative deltas, and the PSX exporter clamps instead of
   wrapping, one side renders fine and the other reads garbage from
   the page. Symmetric breakage (front of head OK, back broken)
   could fit this.

4. **Wrong material on multi-surface mesh.** The FBX MeshInstance3D
   may export multiple surfaces, each wanting a different texture
   in the original Kenney pipeline. Our `material_override` forces
   ALL surfaces to use humanMaleA — surfaces that were authored for
   a different cell of a multi-region atlas would sample the wrong
   region. Check `mesh.get_surface_count()` on the exported
   characterMedium; if >1, we may need per-surface overrides
   instead of `material_override`.

**Quick first diagnostic:**
- Bump `BitDepth=0` → `BitDepth=1` on the skinned mesh in
  `monitor.tscn` line ~1301, re-export, look at the figure again.
  If the body becomes coherent at 8bpp, root cause is #2
  (quantization). If still broken, root cause is #1 or #3.
- Then check `mesh.get_surface_count()` via a one-shot GDScript
  to rule out #4.

**Image to compare against**: source `humanMaleA.png` in
`godot-ps1/assets/monitor/models/kenney_animated-characters-retro/Skins/`.
Open it side-by-side with the in-game screenshot to see which
regions of the atlas the broken triangles are sampling from — that
points directly at the right hypothesis.

     re-run, redeploy to `assets/monitor/textures/decals/`.
   - **Fix path B (post-process)**: post-process the existing PNGs:
     compute alpha bbox, paste centered onto a larger transparent
     canvas, save. Cheaper than regenerating, deterministic, no API
     cost. This is what `gen_menu_assets.py` already does for the
     menu logo via `img.getbbox()` — same pattern works here.

2. **Quad aspect ≠ texture aspect.** Scene authoring used hand-picked
   widths (1.2 × 1.2 for sigils, 0.7 × 1.6 for ghosts). If the
   underlying PNG isn't square / matching aspect, sampling 0..1 UV
   stretches the content asymmetrically and the visible "cut off"
   is actually overflow that landed off-quad.
   - Quick check: list the PNG dimensions of every decal texture
     and compare against the quad widthsx heights authored in
     `place_horror_decals.py` / `place_horror_decals2.py`.

   256×256 page holds 4 4bpp textures laid out 128×256 each (or
   denser packings). If a single decal texture is wider than 128 in
   4bpp, it crosses a page boundary and the runtime samples
   garbage past the boundary. The export log already showed "Kenney

   decals. Verify each decal texture's effective 4bpp width fits

### Audio routing — SPU vs. CDDA vs. XA (open design question)

User raised: *"Should we strictly be using the SPU? Couldn't we use
CDDA or some other form of playing audio?"*

**Short answer**: yes, very much worth it long-term. CDDA is the right
home for music + ambient loops; SPU should keep short SFX. Below is
the breakdown.

**PSX audio paths available:**

| Path | Storage | Memory cost | Channels | Best for |
|------|---------|-------------|----------|----------|
| **SPU (ADPCM)** | SPU RAM, 256 KB cap | each clip lives in RAM | up to 24 | short SFX, anything needing precise trigger / layering |
| **CDDA (Red Book)** | CD audio tracks | **zero** (streamed off disc) | 1 stereo stream | music, long ambient loops |
| **XA-ADPCM** | interleaved on data CD | small ring buffer | up to 8 multiplexed | dialog, multi-channel ambient |
| **SPU streaming** | data sectors → SPU ring | small SPU buffer | 1 sample stream | "voice" tracks, music with custom timing |

**Where each one helps "The Monitor":**

- **`hum_electrical` (25 KB / 4s loop)** — perfect CDDA candidate.
  Seamless loop, deep ambient, ~zero memory cost. Same for any other
  ambient loop (drone, rain, wind, etc.).
- **`shift_end_stinger` (38 KB / 3s)** — borderline. Could be CDDA
  if it doesn't need to layer with SFX during the same playback.
- **`alarm_distant` (38 KB)** — ditto. CDDA frees a lot of RAM.
- **All the small SFX** (clicks, footsteps, creaks, thuds) — stay on
  SPU. They need fast trigger and overlap with each other.

**Catch — development workflow:**
- PCdrv (host filesystem, our current dev loop) **does not play
  CDDA**. CD audio only works off an actual ISO/cue. During dev we'd
  need an SPU fallback for whatever's routed to CDDA, OR accept
  silence on long ambient during PCdrv runs.
- A clean abstraction: each authored audio clip can be tagged
  `Routing = SPU | CDDA | Auto`. Auto = CDDA if available, SPU
  fallback otherwise. Keeps PCdrv usable.

**What it would take to land:**

1. **Runtime**: CDDA play/seek API in `audiomanager.cpp`. PSYQo's
   `CDRom` class already exposes Red Book track playback — wrap it.
2. **Splashpack format**: per-clip routing field + CDDA track number
   list in the header. Format bump (v25).
3. **Exporter**: route clips marked CDDA to mkpsxiso as additional
   audio tracks (Track 02, 03, …) instead of ADPCM-encoding into
   `.spu`. Fall back to SPU for PCdrv runs.
4. **Lua API**: `Audio.PlayCDDA(track, loop)` and route through
   `Audio.PlaySFX(name)` so existing call sites keep working.
5. **mkpsxiso config**: add audio tracks to the CD layout.

**Recommendation**: defer until ISO build (Phase 3 territory). For
the jam deadline, just trim 60 KB of SPU samples; for a polished
post-jam build, CDDA is the right call. Worth a section in
`ROADMAP.md` Phase 3.

**Memory candidate**: "PSX audio routing — SPU for SFX, CDDA for
music/ambient" once the routing system lands. Until then, just keep
this section of the handoff carrying the design notes.
