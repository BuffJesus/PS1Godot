## The drag bug — what it really was

Yesterday's handoff suspected a GTE pipeline hazard: the OFX/OFY
writes looked correct on paper, the raw values logged as expected,
but the mesh "still rendered at screen center (160, 120)." Leads 1–4
all assumed the problem lived at the GTE register level.

**The actual root cause was occlusion, not projection.** A v4
diagnostic that ran `rtps` on the model's local origin right after
our writes and read back `SXY0` proved the GTE was doing exactly what
we asked:

```
UI[0] origin -> SX=262 SY=112 SZ=20480 (expect SX~262 SY~112)
```

The UI preview was always rendering at the dragged pixel. The author
was looking at the **world** `HeadSpider` (world render pass, scene
position), which the intro cutscene rotates — that copy sat at
screen-center regardless of where the UI rect moved, hiding the UI
copy behind it because the world SZ was smaller (closer to camera).

Lesson: when a mesh "doesn't appear to move," confirm which render
pass the user is actually watching before debugging projection math.
The OFX approach would have worked; we switched to TRX/TRY anyway
because it's depth-correct and avoids mutating projection center
mid-frame.

## Architecture updates worth knowing

### UIModel-in-world semantics (`gameobject.hh` bit 5)

`GameObject::isUIModelTarget()` is set at scene init on every
`UIModel.targetObjIndex`. All world render paths skip flagged
objects. Collision / Lua / physics still run — it's a visual-only
flag. Authors who want both a world render AND a HUD preview of
the same mesh duplicate the node into two `PS1MeshInstance` /
`PS1MeshGroup` entries and target the dedicated one.

### OT slot reservation (`renderer.cpp processTriangle forceFrontOT`)

UI polys go into slots `[1, 63]`, world polys into `[65, OT_SIZE)`.
The UI mapping preserves within-model depth ordering via a biased
SZ shift (`(sz - 18000) >> 6`), calibrated against the
`sz0=19915..sz2=20054` sample range observed for a HeadSpider at
`OrbitDistance=5`. If an author sets an unusual orbit distance, UI
polys may clamp to slot 1 or 63 — visible as mild intra-mesh
sorting artifacts. **`UI_SZ_BIAS=18000` is the knob to turn** if
this shows up in practice; a proper fix would derive it from
`currentDistFp12` at runtime. Filed as a known limitation.

### ProjectionH is now auto-derived

`PS1UIModel.ProjectionH` is still an exposed property, but
`SceneCollector` computes the effective value from
`(rectW/2) * gteScaling * orbitDistance / meshHalfExtent` at
export. The manual value is only used as a fallback if the target
mesh's `LocalAabb` can't be resolved (skinned target, unresolved
NodePath). The in-node `[Export]` is left in place for that fallback
but the docstring now names it as such.

The clamp was lifted from `1024` to `65535` (the GTE's uint16
ceiling) since larger rects produce Hs in the 1500–2500 range.

## Known follow-ups

- **`UI_SZ_BIAS` is scene-specific**. Works for typical 3–5m orbit
  distances. A future pass should compute the bias per-UIModel from
  `currentDistFp12` so the depth compression band tracks the author's
  orbit choice.
- **Skinned meshes can't be UIModel targets**. The UI pass skips
  targets with `polyCount == 0`, and skinned GOs zero their polys
  out at init. Currently no warning at export; SceneCollector has a
  "not yet supported as a UIModel target" message for
  PS1MeshGroup-under-PS1MeshGroup but doesn't cover the skinned case.

## Scripting UX — closed, all four tasks landed

All four items from yesterday's "Next focus: scripting experience"
shipped this afternoon. Brief summary of what each provides:

1. **Decimal-literal rewriter** (`fca8b70`) — `SceneCollector.ResolveLuaScript`
   runs every `.lua` source through `LuaDecimalRewriter.Rewrite` before
   byte-encoding. Handles `<digits>.<digits>` cases; skips `.5`, `5.`,
   scientific notation — document-worthy fallbacks, not bugs. 16 unit
   tests in `LuaDecimalRewriterTests`.
2. **Runtime load hint** (`56bfb6a`) — `printLuaLoadError` in `lua.cpp`
   intercepts `L.loadBuffer` failures. If the Lua error contains
   "malformed number" it appends a 4-line fix-suggestion pointing at
   the rewriter + `FixedPoint.newFromRaw` escape hatch. Needle scan is
   a hand-rolled substring helper since `<string.h>` isn't in the
   freestanding MIPS toolchain.
3. **Authoring cheatsheet** (`724febb`) — `docs/lua-ps1-cheatsheet.md`.
   One-pager covering decimal rewriter behavior, +Y-down axis, camera
   head vs player body in third-person, fp12 type taxonomy,
   `IsPressed`-is-JustPressed gotcha, `Debug.Log` cost.
4. **Stub generator** (`26a7f1e`) — `LuaApiStubGenerator` parses the
   `// Namespace.Method(args) -> retval` comment blocks in
   `luaapi.hh` and emits an `---@meta` EmmyLua file at
   `godot-ps1/demo/scripts/_ps1api.lua`. Auto-picked up by Rider since
   same-directory; no config needed. Run via "Regenerate Lua API
   stubs" tools-menu entry. 108 API binds covered. Type inference is
   deliberately shallow (token name heuristics) but the original doc
   comment appears above each stub so authors get real semantics on
   hover. Tests in `LuaApiStubGeneratorTests`.

## Environment reminders

- **Before touching `gameobject.hh` / `splashpack.hh` /
  `scenemanager.hh` / `musicsequencer.hh`**: `make clean && make`
  on the psxsplash side. Makefile doesn't track header deps.
  Stale-.o ABI mismatch manifests as GPU state corruption on the
  first frame.
- **C# plugin rebuild**: `cd godot-ps1 && dotnet build
  PS1Godot.csproj -c Debug`. Godot editor reloads the DLL on next
  tool-menu invocation.
- **PS1 binary rebuild**: `scripts/build-psxsplash.cmd` (incremental).
  For a clean rebuild: `cd psxsplash-main && make clean && cd .. &&
  scripts/build-psxsplash.cmd`.
- **Export + launch**: "Run on PSX" dock button. Always re-exports
  the currently-edited scene before launching PCSX-Redux.
