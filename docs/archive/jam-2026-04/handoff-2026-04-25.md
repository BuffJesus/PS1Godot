### Jam game — Day 5 polish features

#### P1 — Visual events (14 entities + Lua hook)

Added 14 PS1MeshInstance event entities to `monitor.tscn` (Hallway: 4,
Storage: 3, Parking: 4, BackRoom: 3). Each has `StartsInactive=true`,
`Tag=1`, name matches the event id in `feed_manager.lua::EVENTS`.
Three shared materials (`mat_event_dark/wood/light`) and 11 deduped
BoxMesh sub-resources keep VRAM footprint tiny.

`feed_manager.lua::fireEvent` now does `Entity.Find(event.id)` +
`Entity.SetActive(true)` so each event reveals its visual cue on
schedule. Once visible, the entity stays visible for the rest of the
shift — gives the player something to spot if they swap back.

#### P2 — Title card (refactored to separate scene)

Started as a canvas inside `monitor.tscn` with `VisibleOnLoad=true`
and a Lua-driven hide-on-input. **It didn't work cleanly** — see
"Gotchas" below. Refactored into its own `title.tscn` with a tiny
`scene_title_init.lua` controller. Cleanest fix; zero render-order
conflicts.

#### P5 — LOGGED hit/miss badge

`feed_manager.lua::logCurrentMoment` walks EVENTS, checks if any event
on the current feed is within `±window/2` of `shiftFrame`. Green tint
`(90, 220, 90)` on hit, yellow `(230, 200, 80)` on miss. Same
integer-math comparison as `isEventCaught` so psxlua's float-division
quirks can't bite.

#### Per-feed unique floor + wall textures (8 OpenAI-generated)

Each feed (Hallway / Storage / Parking / BackRoom) now has its own

nearest-downsized to 128². Tints reset to `Color(1,1,1,1)` since each

came out too bright on first roll → re-prompted with explicit "DARK" /

`monitor.tscn` rewrites done via Python script: 8 ext_resource

#### Sky — multiple iterations, ended with AI generation

Three attempts:

1. **Procedural** (Python + PIL, sparse stars on dark blue) — looked
   sterile, user wanted AI feel.
2. **OpenAI with "tileable starry texture" prompt** — model produced
   uniform medium-grey (mean RGB 104). Useless; the keyword "tileable"
   confused the model into making a flat plane.
3. **OpenAI without "tileable" + post-process 24-px L↔R cross-fade**
   — proper rich starry sky (mean RGB 0.8/2.6/18, max 255 = bright
   stars). The cross-fade in post guarantees seamless wrap regardless
   of what the model produced.

Final sky lives at `assets/monitor/textures/sky_stars.png` (gitignored).

## Gotchas captured today

Two new memory entries today, both from real debugging time:

1. **`project_psx_ui_sort_order_inverted.md`** — UI canvas sortOrder
   is **LIFO-inverted**. psyqo OT inserts at HEAD, GPU walks
   HEAD→TAIL, painter's algorithm makes the LAST-drawn pixel win — so
   the FIRST-inserted (lowest sortOrder) ends up on top. The inline
   comment in `uisystem.cpp` claims the opposite; it's wrong. Bezel at
   sortOrder 40 has been "on top" because it's the lowest, not the
   highest. **For full-screen overlays (titles, fades): use sortOrder
   0–10.**

2. **`project_scene_init_runs_after_oncreate.md`** — per-object Lua
   `onCreate` runs BEFORE the scene-level Lua's `onSceneCreationEnd`.
   So anything onCreate sets (visibility, state) can be overwritten
   by what onSceneCreationEnd does next. **Bit us when
   feed_manager.onCreate hid the bezel/HUD canvases; the scene-level
   `scene_monitor_init.onSceneCreationEnd` then immediately
   re-showed them.** Lesson: treat scene-level state as authoritative;
   put visibility/audio/control in `onSceneCreationEnd`. For overlays,
   eliminate the conflict via a separate scene.

Plus reinforcement of two existing memory items:

3. **Godot DLL hot-reload trap** — surfaced again when an export
   reported "wrote 170 bytes, expected 168" header mismatch despite
   the source code already saying `pad_sky` was 1 byte. Restart Godot
   to reload plugin DLL after `dotnet build`.

4. **Stale-`.o` ABI trap** — header changes to runtime structs (added
   `m_skyScrollFrame` to Renderer) need `make clean && make`. Bare
   `make` fails silently with garbage.
