## The long Lua editor arc (worth reading first)

Syntax highlighting took ~4h of surprise. Notes to save future-you:

1. **`ScriptLanguageExtension` registers the language; `EditorSyntaxHighlighter`
   registers the colors.** These are two separate Godot classes. Our
   `PS1LuaScriptLanguage` (GDExtension) registers `.lua` as a script
   language and drives delimiter-based comment/string coloring. But
   keyword coloring comes from a DIFFERENT mechanism — an
   `EditorSyntaxHighlighter` subclass registered via
   `ScriptEditor::register_syntax_highlighter`. `ScriptLanguage::_get_reserved_words`
   is read by the *parser*, not the editor.
2. **C# subclasses of `EditorSyntaxHighlighter` don't register properly.**
   We confirmed via trace: `RegisterSyntaxHighlighter(...)` returns without
   error, but `_GetSupportedLanguages` / `_GetLineSyntaxHighlighting`
   are never queried back. The GDExtension path works fine — so the
   highlighter class has to live in `ps1lua.gdextension`.
3. **`EditorSyntaxHighlighter` parent class doesn't exist at
   `MODULE_INITIALIZATION_LEVEL_SCENE`.** It materializes at
   `MODULE_INITIALIZATION_LEVEL_EDITOR`. `GDREGISTER_CLASS` for our
   highlighter has to run at EDITOR level.
4. **`ScriptEditor::get_script_editor()` returns null at EDITOR-level
   init.** Even at EDITOR level, the script editor isn't constructed
   yet. Workaround: register the CLASS in C++ (at EDITOR init), but
   instantiate + register the INSTANCE from the C# plugin's
   `_EnterTree` which fires later. That split is what `47af3d4` and
   `b798f63` together implement.
5. **Godot 4.7 requires ~20 virtual overrides** on `ScriptLanguageExtension`
   before the editor stops erroring. `_lookup_code` is one; also
   `_complete_code`, `_create_script`, `_preferred_file_name_casing`,
   every `_profiling_*`, `_add_global_constant`, `_debug_get_stack_level_instance`.
   All stubbed minimally. `_lookup_code` specifically needs both
   `"result"` and `"type"` keys or Godot logs `ERR_UNAVAILABLE`.
6. **Per-tab highlighter cache gotcha.** If a `.lua` file was ever
   opened in Godot before `ps1lua.gdextension` landed, its tab state
   remembers "Plain Text" forever. Author picks "PS1 Lua" from the
   highlighter dropdown once → Godot persists the choice. Documented
   in `docs/lua-editor-setup.md`.

Autocomplete (`47af3d4`) then added:

- Python `gen_api_data.py` parses `luaapi.hh` at build time → emits
  `src/ApiData.gen.cpp` with a static array of 92 entries
  (namespace, name, args, return, docstring).
- `_complete_code` scans backward from Godot's `￿` cursor-sentinel
  to detect context (`ns.prefix` vs bare prefix vs inside a call),
  serves matching entries.
- Each option dict needs `font_color` key or Godot 4.7 logs errors per
  option (noisy but non-fatal).

## Disparities to fix tomorrow (ranked)

### 1. **Stub generator drops 22 API entries** (small, quick win)

`luaapi.hh` has **114** signature-shaped comment lines; `gen_api_data.py`
picks up **92**. Missing 22 are likely:

- Return types with parenthesized clarifiers: `// Entity.GetRotationY(object) -> number (radians)`
  — the regex's `.+?` + `\s*$` should handle these but something's off. Inspect.
- Signatures that span multiple lines (`Entity.Spawn(tag, {x,y,z} [, rotY]) -> …`
  split across two lines) — less common but possible.
- Lines where the trailing `->` retval has a comma or other quirk that the
  lazy quantifier can't handle.

Fix: rerun gen_api_data.py against the header, diff the output against
a loose-regex list, debug the non-matches. Expect 30-60 min of work.

### 2. **Lua keyword completion** (trivial, 15 min)

Today's autocomplete only suggests from the API table. Typing `fu` doesn't
suggest `function`. Add a static list of Lua keywords alongside the
namespace list in `_complete_code`:

```cpp
static const char *LUA_KEYWORDS[] = {
    "and", "break", "do", "else", "elseif", "end", "false", "for",
    "function", "goto", "if", "in", "local", "nil", "not", "or",
    "repeat", "return", "then", "true", "until", "while", "self",
};
```

Return them as `KIND_CLASS` (or a hypothetical KIND_KEYWORD if one
exists — check `CodeEdit::CodeCompletionKind`). Filter by prefix match.

### 3. **Event-handler name completion** (small, 20 min)

Inside `function on...` at top level, suggest the known PS1Godot event
names: `onCreate`, `onUpdate`, `onCollide`, `onInteract`,
`onTriggerEnter`, `onTriggerExit`, `onDestroy`, `onEnable`,
`onDisable`, `onButtonPress`, `onButtonRelease`. These are all
hardcoded in `psxsplash-main/src/lua.cpp`
(`eventMask |= EVENT_ON_*` lines 473-483) — scrape them into
a second static list + trigger when context is "line starts with
`function on`".

### 4. **Local-variable harvest** (medium, 60-90 min)

Once 1-3 are in, the real authoring pain is that `local foo = ...` in
your script doesn't show up in completion. A naive regex scan
(`^\s*local\s+(\w+)` and `^\s*(\w+)\s*=\s`) gets 80% of the way there
for non-block contexts. Scope is fuzzy (we'd show all locals
everywhere), but that's still better than zero.

### 5. **Call-hint clipping at end-of-script** (not fixable from extension)

When the cursor is on the last line, Godot draws the call-hint popup
*below* it — off-screen. Not addressable from `ScriptLanguageExtension`;
it's a Godot editor UI decision. Workaround documented in
`lua-editor-setup.md`: leave a trailing blank line. Close as "won't
fix" unless we find a Godot editor setting that flips hint position.

### 6. **`psxsplash-main/third_party/nugget` submodule drift**

`git status` has flagged this since before today. Untracked content
inside the submodule. Still unresolved — worth investigating quickly
to de-noise `git status`. Probably either (a) a nested submodule we
should init, or (b) build artifacts that should be gitignored in the
submodule.

## psyqo capabilities we're leaving on the table

Surveyed today. psxsplash's coverage of psyqo is solid for rendering +
math + audio basics. The actual gaps are (a) a handful of psyqo
headers the runtime never touches, and (b) plenty of capability psyqo
exposes that isn't wired through to Lua. Ranked by author impact:

1. **`Debug.DrawLine(a, b, color)` / `Debug.DrawAABB(min, max)`** —
   `psyqo/primitives/lines.hh` ships `Line`, `LineG2`, `LineG3`. We
   emit only triangles + sprites + rects today. A runtime line
   submission surface + Lua binding is ~50 lines of C++. Instant
   gameplay-debugging superpower (raycast paths, nav polygons, AABB
   wireframes all visible in-game). **~1–2h.**
2. **GPU scissor → `UI.Clip(rect)`** — `psyqo/gpu.hh` has
   `enableScissor` / `getScissor` / `Prim::Scissor`. Unused.
   Exposing it fixes today's UI-model-pokes-outside-its-rect
   cosmetic (no more `UI_SZ_BIAS` tuning), and enables general UI
   masks (HP bars fill inside a frame, cropped portraits, etc.).
   **~2–4h.** Likely splashpack-format bump for per-element clip
   flags.
3. **Bezier-driven cutscenes** — `psyqo/bezier.hh` is full curve
   eval; unused. `PS1AnimationTrack` could grow a `BezierPosition`
   track type, cutscenes could have smooth camera rails instead of
   linear keyframe interp. **~4–6h.** Format bump.
4. **Per-pad input** — `advancedpad.hh` handles pad 0..3 + multitap;
   our Lua `Input.IsPressed(button)` assumes pad 0. Bind as
   `Input.IsPressed(pad, button)` with default pad=0 for backward
   compat. Enables 2P. **~1h.**
5. **Native C++20 coroutines → `Task.RunOverFrames(fn, budget_us)`** —
   `psyqo/coroutine.hh` is PS1-native coroutines; unused by psxsplash
   entirely. Lua has its own `coroutine` via psxlua but no frame
   budget. Add a C++ scheduler that yields mid-function when the
   budget runs out. ROADMAP Phase 2.5 `[runtime]` item —
   pre-existing scope. **Medium (~half-session)**.
6. **Runtime mesh submission — `Mesh.Submit(verts, tris, tpage, aabb)`** —
   ROADMAP Phase 2.5 `[runtime]` item. All GPU primitive pieces are
   in psyqo. Enables voxel/procedural/dynamic decals. **Bigger
   (1+ session)**, architectural.
7. **Dynamic VRAM upload → `Texture.WriteRegion(pageXY, data)`** —
   `gpu.hh::uploadToVRAM`. For portrait swaps, runtime decals, mask
   textures. Not in Lua.
8. **More GTE kernels** — `ncs`/`ncds`/`dpcs`/`sqr` etc. Psyqo has
   ~30 kernels, we use 3 (`rtpt`, `rtps`, `nclip`). Matters only if
   we want PSX-native gouraud lighting from the GTE; we roll our
   own lighting in psxsplash. **Low-priority unless rewriting the
   lighting pipeline.**
9. **SPU per-voice mid-playback control** — `psyqo/spu.hh` itself is
   actually thin (no pitch bend / reverb API on active voice). Would
   need to go below psyqo to the raw SPU registers. Deferred.

## Strategic — is the plugin-vs-fork bet still right?

Asked twice today. Two distinct forks of that question:

### Fork-for-dev-access (would we want deeper Godot internals?)

No. Today's 4h of ScriptLanguageExtension friction was one-time and
now documented. A fork costs perpetual upstream-merge tax and
distribution friction, for zero actual PSX-authoring capability we
can't already hit from the plugin surface (rendering runs on
psxsplash, scripting runs on psxlua, export format is ours). None
of today's pain was engine-level; it was all Godot-API discovery.
Plugin stays.

### Fork-for-focused-UX (general-purpose editor vs. PSX-only toolkit)

Separate question. The *current* UX is plainly "Godot with plugins"
rather than "a PSX authoring tool" — no viewport aspect lock, no
default-wrong project settings blocked, GDScript still offered as
primary, 2D editor tab still visible, inspectors still show
PBR/Bullet/etc. properties irrelevant to PSX. That IS worth fixing.

But the fix isn't a fork — it's heavier use of the EditorPlugin API
we've only touched lightly. Documented as **Phase 3 — Aggressive
Editor UX track** in ROADMAP (to add). Top candidates:

- `EditorPlugin.set_main_screen_editor` to register a "PS1" top-bar
  tab that replaces the default 2D/3D/Script workflow with a
  PSX-focused dashboard
- Project-setting locks applied on plugin enable: viewport 320×240,
  aspect 4:3, rendering method "compatibility", disable
  auto-reimport of texture formats we don't use
- Custom `EditorInspectorPlugin` that hides PBR / Bullet / post-FX
  properties from mesh / material inspectors (they're visual noise
  for us)
- Import postprocessor that defaults textures to 256×256 cap, 4bpp
  / 8bpp CLUT palette, no mipmaps
- Scene-create template "PS1 Scene" that drops in a pre-configured
  root with the right children + project settings
- Remove / rename tool-menu entries that don't apply
- Custom editor theme with PSX-era palette on plugin enable
- VRAM / SPU / triangle budgets in a persistent status-bar HUD
  (not tucked in the right dock)

We've done maybe 10% of that list. 80% of the "feels like a
dedicated PSX tool" experience lives in this lane, achievable
without fork tax. Worth adding as explicit roadmap scope.

**Decision stays:** plugin + GDExtension. Re-open the fork question
only if we hit a specific wall the EditorPlugin API can't route
around. None hit today.
