## Next task — animate the player with Walking.fbx

**Goal:** the player avatar plays a walk-cycle animation while moving,
idles (static pose or breathing) while standing. Same pipeline we used
for the test cylinder's wave animation, now applied to the real player
mesh.

### Concrete steps

1. **Inspect Walking.fbx in Godot's import panel.**
   - FileSystem → select `Walking.fbx` → Import tab.
   - What's in it? Bones, weighted mesh, one or more animation
     clips (probably a walk cycle, maybe an idle + walk).
   - Is the mesh skinned with proper bone weights? If it's an auto-rig
     output (Mixamo, etc.) it usually is.
   - Bone count matters — our runtime caps are noted in
     `docs/ps1_large_rpg_optimization_reference.md` and
     `psxsplash-main/src/skinmesh.hh` (see `MAX_BONES_PER_MESH`).
     Mixamo humanoids typically ship with ~65 bones; we may need to
     retarget / reduce.

2. **Convert the mesh to PS1SkinnedMesh.**
   - In `demo.tscn`, replace the static `Player` MeshInstance3D under
     `PS1Player` with a `PS1SkinnedMesh` node referencing the imported
     humanoid mesh + its skeleton.
   - `PS1SkinnedMesh` docs are in
     `addons/ps1godot/nodes/PS1SkinnedMesh.cs`. The test cylinder in
     `SkinnedTest` subtree (demo.tscn ~line 770) is the reference
     wiring.
   - **Watch for the gotcha** in
     `memory/project_skinned_mesh_gotchas.md`: unweighted verts silently
     default to bone 0. Every vert must have a bone assignment or
     split the mesh.

3. **Bake the animation clips.**
   - The exporter's skin pipeline bakes `AnimationPlayer` clips at
     author-set FPS. Set FPS to ~15 (PSX is 30fps; 15 gives a good
     balance of smoothness vs splashpack size).
   - Name clips `walk` and `idle` (or just `walk` if the FBX only has
     one).

4. **Drive the animation from Lua.**
   - In `test_logger.lua` onUpdate, detect player movement:
     ```lua
     local p = Player.GetPosition()
     local moved = (p.x ~= lastPlayerX) or (p.z ~= lastPlayerZ)
     if moved and not isWalking then
         SkinnedAnim.Play("Player", "walk", { loop = true })
         isWalking = true
     elseif not moved and isWalking then
         SkinnedAnim.Play("Player", "idle", { loop = true })
         isWalking = false
     end
     lastPlayerX, lastPlayerZ = p.x, p.z
     ```
   - Or cleaner: check `Input.GetAnalog(Input.LEFT_STICK)` magnitude
     instead of position delta (doesn't drift on frame-time variance).

5. **Smoke-test on PSX.**
   - Run the scene. The humanoid should walk-cycle as you move the
     stick and stop moving its legs when you release. If the walk
     animation looks T-posed or missing limbs, it's a bone-weighting
     or bone-count issue. Check the export log for
     `[PS1Godot] PS1SkinnedMesh 'Player': N bones, X triangles, K
     baked clips` and verify N matches what's in the FBX.

### Known risks

- **Bone count over the runtime cap.** Mixamo rigs run 60+ bones; our
  cap is lower (check `MAX_BONES_PER_MESH`). Either retarget to fewer
  bones in Blender before importing, or bump the runtime cap and
  re-baseline the memory budget.
- **Per-vertex rigid binding vs weighted skinning.** PSX can't do
  blended bone weights — each vertex gets exactly one bone. If the
  humanoid was authored with 4-bone weighted skinning, the exporter
  picks the highest-weighted bone per vertex, which may look bad at
  joints. Known limitation from bullet 11.
- **FBX import root-motion handling.** Godot's FBX importer sometimes
  puts the animation on the root bone's transform (root motion);
  other times it's on child bones. If the walk cycle makes the
  character drift forward on screen (adding to player movement),
  that's root motion — needs to be stripped in the importer or
  Blender-retargeted in-place.
